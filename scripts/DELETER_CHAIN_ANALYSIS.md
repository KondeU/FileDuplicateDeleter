## 二、选择保留文件的策略链（核心）

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         select_file_to_keep() 函数                           │
└─────────────────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
                        ┌───────────────────────┐

                        │  初始候选: all indices   │
                        └───────────────────────┘
```

## 三、祖先链构建详解

### 3.1 祖先链方向设计

```
旧设计（问题）: 从文件向上到 root
┌─────────────────────────────────────────────────────────────────┐
│  file: D:\Photos\A\B\file.jpg                                   │
│                                                                  │
│  祖先链: [(A/B), (A), (root)]  ← 从深到浅                        │
│                                                                  │
│  问题: 不同深度文件按数组索引比较时                                │
│        depth 1 可能是 root（浅文件）vs A（深文件）                │
│        → 比较的是不同层级！                                       │
└─────────────────────────────────────────────────────────────────┘

新设计（正确）: 从 root 向下到文件
┌─────────────────────────────────────────────────────────────────┐
│  file: D:\Photos\A\B\file.jpg                                   │
│  root: D:\Photos                                                │
│                                                                  │
│  祖先链: [(root), (A), (A/B)]  ← 从浅到深                        │
│                                                                  │
│  优势: 相同 depth 总是对应相同层级                                │
│        depth 1 总是 root 的子目录层级                            │
│        → 公平比较                                                │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 祖先链生成过程

```python
def get_ancestral_mtime_from_root(filepath, root_dir):
    # 1. 从文件目录向上收集路径
    path_chain = []
    current = file_path.parent
    while current != root_path:
        path_chain.append(current)
        current = current.parent
    
    # 2. 反转：从 root 向下
    path_chain = path_chain[::-1]
    
    # 3. 获取每个目录的 mtime
    result = [(path, os.path.getmtime(path)) for path in path_chain]
    
    return result
```

|---|----------|----------|----------|------|

## 五、深度差异处理策略详解

### 5.1 为什么选择保守策略？

```
场景: file1(root/A/a.jpg) vs file2(root/B/C/a.jpg) vs file3(root/B/D/a.jpg)

假设:
  - mtime(A) = mtime(B)  → 策略2无法区分，候选=[file1, file2, file3]
  - 祖先链:
    - file1: [(root), (A)]           长度=2
    - file2: [(root), (B), (B/C)]    长度=3
    - file3: [(root), (B), (B/D)]    镀度=3

策略3执行:
  depth 1: unique_dirs={A, B}, mtime(A)=mtime(B) → candidates=[全部]
  depth 2: file1 长度=2 < depth=2 → all_have_this_depth=False → break

结果: 进入策略4（用户选择）

问题: file2 和 file3 在 depth 2 本来可以比较（B/C vs B/D），但被 file1 拖累

为什么选择保守策略?
  1. file1 可能是用户想保留的"原始文件"（位置更接近 root）
  2. 如果排除 file1 继续比较，可能自动删除了用户想要的文件
  3. 让用户选择是安全的，即使增加了一点交互成本
```

### 5.2 替代方案对比

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **当前方案** | 有短链就停止 | 保守安全 | 可能不够智能 |
| 方案A | 排除短链继续 | 更智能 | 可能误删用户想要的文件 |
| 方案B | 短链用最小mtime | 短链自动淘汰 | 逻辑复杂，用户可能困惑 |

## 六、删除前验证详解

### 6.1 三层验证机制

```
┌─────────────────────────────────────────────────────────────────┐
│  验证层 1: 文件存在性                                             │
│  ─────────────────────────────────────────────────────────────  │
│  检查点: os.path.exists(filepath)                               │
│  失败处理: [跳过] file_skipped += 1                              │
│  目的: 处理文件在扫描后被删除的情况                               │
└─────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────┐
│  验证层 2: 哈希一致性                                             │
│  ─────────────────────────────────────────────────────────────  │
│  检查点: recompute MD5+SHA256 == report_hash                     │
│  失败处理: [跳过] verify_failed += 1                             │
│  目的: 检测文件在扫描后被修改                                     │
└─────────────────────────────────────────────────────────────────┘
                                    │
                                    ▼
┌─────────────────────────────────────────────────────────────────┐
│  验证层 3: 逐字节比对                                             │
│  ─────────────────────────────────────────────────────────────  │
│  检查点: byte-by-byte compare (file vs keep_file)                │
│  失败处理: [跳过] verify_failed += 1                             │
│  目的:                                                          │
│  - 检测哈希碰撞（极罕见）                                        │
│  - 检测保留文件被修改（race condition）                          │
│  - 最终确认真正一致                                              │
└─────────────────────────────────────────────────────────────────┘
```

### 6.2 验证顺序的意义

| 验证顺序 | 原因 |
|----------|------|
| 先检查存在性 | 后续验证需要文件存在 |
| 再检查哈希 | 哈希计算比逐字节快 |
| 最后逐字节 | 最可靠但最慢，作为最终确认 |

### 6.3 逐字节比对实现

```python
def verify_files_identical(file1, file2):
    # 快速检查: 文件大小
    if os.path.getsize(file1) != os.path.getsize(file2):
        return False
    
    # 逐块比对（64KB chunks）
    with open(file1, "rb") as f1, open(file2, "rb") as f2:
        while True:
            chunk1 = f1.read(65536)
            chunk2 = f2.read(65536)
            if chunk1 != chunk2:
                return False
            if not chunk1:  # EOF
                return True
```

## 七、异常场景完整列表

### 7.1 文件层面异常

| # | 异常场景 | 触发条件 | 处理方式 | 对用户影响 |
|---|----------|----------|----------|------------|
| 1 | 文件不存在 | os.path.exists() == False | 跳过，计入 skipped | 安全，不误删 |
| 2 | 文件无读取权限 | PermissionError | 哈希验证失败，跳过 | 安全，不误删 |
| 3 | 文件被锁定 | IOError | 哈希验证失败，跳过 | 安全，不误删 |
| 4 | 文件被修改（扫描后） | 哈希不一致 | 验证失败，跳过 | 安全，保留修改后版本 |
| 5 | 文件被截断 | size变化 | 逐字节比对失败 | 安全，跳过 |
| 6 | 文件内容替换 | 哈希不一致或字节不一致 | 验证失败 | 安全，不误删 |
| 7 | 空文件 | size=0 | 正常处理 | 保留空文件副本 |

## 八、Race Condition 分析

### 8.1 可能的时间窗口问题

```
时间线分析:
───────────────────────────────────────────────────────────────────
T1: analyzer 扫描 → 计算哈希 → 写入报告
    │
    │   ⚠ 时间窗口: 用户可能在此期间修改文件
    │
T2: deleter 启动
    │
    ├─ 验证层1: 检查文件存在性
    │       │
    │       │   ⚠ 时间窗口: 文件可能在此期间被删除
    │       │
    ├─ 验证层2: 重新计算哈希
    │       │
    │       │   ⚠ 时间窗口: 文件可能在此期间被修改
    │       │
    ├─ 选择保留文件
    │       │
    │       │   ⚠ 时间窗口: 保留文件可能在此期间被修改/删除
    │       │
    ├─ 验证层3: 逐字节比对
    │       │
    │       │   ⚠ 时间窗口: 最危险！
    │       │              f1.read() 后 f2.read() 前，
    │       │              文件可能被修改
    │       │
    ├─ 执行删除
    │
T3: 完成
───────────────────────────────────────────────────────────────────
```

### 8.2 最坏情况分析

```
最坏情况:
───────────────────────────────────────────────────────────────────
1. 逐字节比对中:
   - f1.read(chunk1)  → chunk1 = "原始内容"
   - ⚠ 此时 f1 被其他进程修改
   - f2.read(chunk2)  → chunk2 = "新内容"
   - chunk1 != chunk2 → 比对失败
   
2. 结果:
   - 比对失败 → 跳过删除
   - 安全！不会误删

结论: 即使有 race condition，也不会导致误删
      最坏情况只是跳过删除，需要用户手动处理
───────────────────────────────────────────────────────────────────
```

### 9.1 三种模式对比

## 十二、总结

### 12.1 安全机制汇总

| 机制 | 实现方式 | 保护效果 |
|------|----------|----------|
| 多层验证 | 存在→哈希→逐字节 | 防止误删 |
| 哈希重验 | 删除前重新计算 | 检测修改 |
| 逐字节比对 | 最终确认 | 防止哈希碰撞 |
| 回收站默认 | recycle 模式 | 可恢复 |
| dry-run | 模拟模式 | 预览 |
| 用户选择 | 策略4 | 无法自动区分时人工干预 |
| 详细日志 | 每组记录 | 可追溯 |
| 深度差异保护 | break → 策略4 | 保守策略 |

### 12.2 核心设计原则

1. **安全优先**: 宁可跳过/让用户选择，也不自动做出可能错误的决定
2. **多层验证**: 三层验证确保真正一致才删除
3. **可追溯**: 详细日志记录每个决策
4. **保守策略**: 深度差异时交给用户，不自动排除
5. **默认安全**: recycle 为默认模式，send2trash 不可用时报错而非静默降级
